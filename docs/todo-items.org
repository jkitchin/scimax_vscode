* ‚úÖ TODO Items

Scimax VS Code provides a comprehensive task management system built into your
documents. TODO items are headlines with a special keyword that indicates their
status.

* ‚úÖ Basic TODO Functionality

** ‚úÖ Creating TODO Items

Add TODO after the headline stars to create a task:

#+BEGIN_SRC org
,* TODO Write documentation
,** TODO Complete first draft
,** TODO Review and edit
#+END_SRC

** ‚úÖ Cycling TODO States

Press Ctrl+C Ctrl+T on a headline to cycle through TODO states:

#+BEGIN_EXAMPLE
(no keyword) ‚Üí TODO ‚Üí DONE ‚Üí (no keyword)
#+END_EXAMPLE

Or use the command scimax.org.cycleTodo.

With speed commands enabled, press t at the beginning of a headline.

** ‚úÖ TODO Keywords

The default TODO keywords are:

| Keyword | Meaning                    |
|---------+----------------------------|
| TODO    | Task that needs to be done |
| DONE    | Completed task             |

Additional states available:

| Keyword     | Meaning                              |
|-------------+--------------------------------------|
| NEXT       | Next action to take                  |
| WAITING    | Waiting for external input           |
| HOLD       | On hold / paused                     |
| CANCELLED  | Task cancelled (won't be done)       |

** ‚ö†Ô∏è Logging Completion

When you mark a task as DONE, a CLOSED timestamp is automatically added:

#+BEGIN_SRC org
,* DONE Write introduction
CLOSED: [2026-01-13 Mon 15:30]
#+END_SRC

* ‚úÖ Priorities

Priorities help you identify the most important tasks.

** ‚úÖ Setting Priorities

Add a priority cookie after the TODO keyword:

#+BEGIN_SRC org
,* TODO [#A] Urgent and important task
,* TODO [#B] Important but not urgent
,* TODO [#C] Nice to have
#+END_SRC

** ‚úÖ Priority Levels

| Priority | Meaning                         |
|----------+---------------------------------|
| [#A]     | High priority - do first        |
| [#B]     | Medium priority - do soon       |
| [#C]     | Low priority - do when possible |

** ‚úÖ Priority Commands

| Key/Command                | Action                    |
|----------------------------+---------------------------|
| Speed key 1               | Set priority A            |
| Speed key 2               | Set priority B            |
| Speed key 3               | Set priority C            |
| Speed key 0               | Remove priority           |
| Speed key ,               | Increase priority (C‚ÜíB‚ÜíA) |

You can also set priority through the command palette:
scimax.org.setPriority

* ‚úÖ Progress Indicators

Track progress through subtasks using statistics cookies and checkboxes.

** ‚úÖ Statistics Cookies [/]

*** ‚ö†Ô∏è Fraction Format

Shows completed/total count:


**** Project Tasks [3/4]
***** ‚ö†Ô∏è Task 1
***** ‚úÖ Task 2
***** ‚úÖ Task 3
***** ‚úÖ [#C] Task 4


*** ‚ö†Ô∏è Percentage Format

Shows completion percentage:

**** Project Progress [25%]
***** DONE Design
***** DONE Implementation
***** ‚úÖ Testing
***** TODO Deployment

Statistics cookies update automatically when child TODO states change.

*** ‚úÖ Checkbox Lists

For simpler task lists within a section, use checkboxes:

* Shopping List [2/4]
- [X] Milk
- [X] Bread
- [ ] Eggs
- [x] Butter


*** ‚úÖ Checkbox States

| Syntax | State                                   |
|--------+-----------------------------------------|
| [ ]    | Unchecked / not done                    |
| [X]    | Checked / done                          |
| [-]    | Partially complete (has mixed children) |

*** ‚úÖ Toggling Checkboxes

Press Ctrl+C Ctrl+C with cursor on a checkbox line to toggle it.

Or use scimax.org.toggleCheckbox.

**** ‚úÖ Nested Checkboxes

Checkboxes can be nested. Parent checkboxes show aggregate state:

#+BEGIN_SRC org
,* Packing List [1/2]
- [-] Clothes [2/3]
  - [X] Shirts
  - [X] Pants
  - [ ] Socks
- [X] Toiletries [3/3]
  - [X] Toothbrush
  - [X] Soap
  - [X] Towel
#+END_SRC

*** ‚úÖ Inserting Checkboxes

| Command                   | Description               |
|---------------------------+---------------------------|
| scimax.org.insertCheckbox | Insert checkbox at cursor |

Type - [ ] followed by space to manually create a checkbox item.

* ‚úÖ Extended TODO Keywords

Beyond TODO and DONE, use these states for workflow management:

** ‚úÖ NEXT

Indicates the very next action to take:

#+BEGIN_SRC org
,* NEXT Call client about project requirements
#+END_SRC

Use NEXT for tasks you're actively working on or will work on immediately.

** ‚úÖ WAITING

Task is blocked waiting for someone or something:

#+BEGIN_SRC org
,* WAITING Review from manager
:PROPERTIES:
:WAITING_ON: John Smith
:END:
#+END_SRC

** ‚úÖ HOLD

Task is paused but not cancelled:

#+BEGIN_SRC org
,* HOLD Website redesign
Paused until Q2 budget approval.
#+END_SRC

** ‚úÖ CANCELLED

Task won't be done (but kept for records):

#+BEGIN_SRC org
,* CANCELLED Old feature request
No longer relevant after product pivot.
#+END_SRC

** ‚úÖ Workflow Example

A typical task progression:

#+BEGIN_EXAMPLE
TODO ‚Üí NEXT ‚Üí WAITING ‚Üí NEXT ‚Üí DONE
         ‚Üì
       HOLD ‚Üí NEXT ‚Üí DONE
         ‚Üì
     CANCELLED
#+END_EXAMPLE

* ‚úÖ Custom TODO Keywords

You can define your own TODO workflow states for each file using the #+TODO:
keyword. This is useful for project-specific workflows like code review,
content publishing, or approval processes.

** ‚úÖ Defining Custom States

Add #+TODO: at the top of your org file (before any headings):

#+BEGIN_SRC org
,#+TODO: DRAFT REVIEW | PUBLISHED REJECTED
#+END_SRC

The | separates active (incomplete) states from done (complete) states.
When you cycle TODO states, they will follow this sequence:

#+BEGIN_EXAMPLE
(none) ‚Üí DRAFT ‚Üí REVIEW ‚Üí PUBLISHED ‚Üí (none)
#+END_EXAMPLE

** ‚úÖ Multiple TODO Lines

You can define multiple workflow sequences:

#+BEGIN_SRC org
,#+TODO: TODO IN-PROGRESS | DONE
,#+TODO: BUG INVESTIGATING FIXING | FIXED WONTFIX
#+END_SRC

All states from all #+TODO: lines are recognized.

** ‚úÖ Without Pipe Separator

If you omit the |, the last state is treated as done:

#+BEGIN_SRC org
,#+TODO: IDEA DRAFT REVIEW PUBLISHED
#+END_SRC

This makes PUBLISHED the done state.

** ‚úÖ Syntax Variants

These keywords are all equivalent:

| Keyword       | Usage                              |
|---------------+------------------------------------|
| #+TODO:      | Standard TODO keyword definition   |
| #+SEQ_TODO:  | Sequential workflow (same as TODO) |
| #+TYP_TODO:  | Type-based workflow                |

** ‚úÖ Example Workflows

*** ‚úÖ Code Review Workflow

#+BEGIN_SRC org
,#+TODO: DRAFT REVIEW APPROVED | MERGED REJECTED

,* DRAFT Implement new feature
,* REVIEW Fix login bug
,* APPROVED Update documentation
#+END_SRC

*** ‚úÖ Content Publishing Workflow

#+BEGIN_SRC org
,#+TODO: IDEA OUTLINE WRITING EDITING | PUBLISHED ARCHIVED

,* WRITING Blog post about org-mode
,* EDITING Tutorial on source blocks
#+END_SRC

*** ‚úÖ Project Management Workflow
#+BEGIN_SRC org
,#+TODO: BACKLOG SPRINT IN-PROGRESS TESTING | DONE CANCELLED

,* IN-PROGRESS User authentication
,* TESTING Payment integration
#+END_SRC

* ‚úÖ Logging State Changes

Scimax VS Code can log when and how TODO states change.

** ‚ö†Ô∏è LOGBOOK Drawer

State changes are recorded in the LOGBOOK drawer:

#+BEGIN_SRC org
,* DONE Complete documentation
CLOSED: [2026-01-13 Mon 16:00]
:LOGBOOK:
- State "DONE"       from "TODO"       [2026-01-13 Mon 16:00]
- State "TODO"       from "WAITING"    [2026-01-12 Sun 10:00]
- State "WAITING"    from "TODO"       [2026-01-10 Fri 14:30]
:END:
#+END_SRC

** ‚ö†Ô∏è Timestamps on State Changes

When enabled, each state change adds:
- A timestamp showing when the change occurred
- The previous state

This creates a complete history of task progress.

* ‚ö†Ô∏è Tags for TODO Items

Tags provide additional categorization beyond the headline hierarchy.

** ‚ö†Ô∏è Adding Tags

Tags appear at the end of the headline, enclosed in colons:

#+BEGIN_SRC org
,* TODO Review quarterly report                               :work:urgent:
,* TODO Buy groceries                                         :personal:errands:
#+END_SRC

** ‚ö†Ô∏è Tag Commands

| Key/Command                | Action                    |
|----------------------------+---------------------------|
| Speed key Shift+;         | Edit tags                 |
| scimax.org.setTags        | Set tags via command      |

** ‚ö†Ô∏è Inherited Tags

Child headlines inherit their parent's tags:

#+BEGIN_SRC org
,* Work Projects                                              :work:
,** TODO Project A                                            :client1:
This task has both :work: and :client1: tags.
,** TODO Project B                                            :internal:
This task has both :work: and :internal: tags.
#+END_SRC

** ‚ö†Ô∏è Common Tag Conventions

| Tag         | Usage                                 |
|-------------+---------------------------------------|
| @office    | Context: at office                    |
| @home      | Context: at home                      |
| @computer  | Context: needs computer               |
| @phone     | Context: phone call                   |
| urgent     | Priority indicator                    |
| project    | Part of larger project                |

** ‚ö†Ô∏è Filtering by Tags

Use agenda views or search to filter tasks by tag:

- scimax.db.searchByTag - Search for specific tags
- scimax.agenda.filterByTag - Filter agenda by tag

* ‚ö†Ô∏è Scheduling and Deadlines

Assign dates to tasks for planning.

** ‚ö†Ô∏è SCHEDULED

When you plan to start or work on a task:

#+BEGIN_SRC org
,* TODO Write report
SCHEDULED: <2026-01-15 Wed>
#+END_SRC

** ‚ö†Ô∏è DEADLINE

When a task must be completed by:

#+BEGIN_SRC org
,* TODO Submit proposal
DEADLINE: <2026-01-20 Mon>
#+END_SRC

** ‚ö†Ô∏è Both Together

#+BEGIN_SRC org
,* TODO Complete project
SCHEDULED: <2026-01-15 Wed> DEADLINE: <2026-01-25 Sat>
#+END_SRC

** ‚ö†Ô∏è Adding Dates

| Speed Key | Action                               |
|-----------+--------------------------------------|
| s        | Add/edit SCHEDULED date              |
| d        | Add/edit DEADLINE date               |
| .        | Insert timestamp                     |

Or use commands:
- scimax.org.schedule
- scimax.org.deadline

See [[file:timestamps.org][Timestamps]] for more on date formats.

* ‚ö†Ô∏è Time Tracking

Track time spent on tasks using clock entries.

** ‚ö†Ô∏è Clocking In/Out

| Speed Key  | Action                              |
|------------+-------------------------------------|
| Shift+I   | Clock in (start timing)             |
| Shift+O   | Clock out (stop timing)             |

** ‚ö†Ô∏è Clock Entries

Clock entries are stored in the LOGBOOK drawer:

#+BEGIN_SRC org
,* TODO Working on documentation
:LOGBOOK:
CLOCK: [2026-01-13 Mon 09:00]--[2026-01-13 Mon 10:30] =>  1:30
CLOCK: [2026-01-12 Sun 14:00]--[2026-01-12 Sun 16:00] =>  2:00
:END:
#+END_SRC

** ‚ö†Ô∏è Total Time

The total clocked time appears in the properties or can be calculated for
reports.

* ‚ö†Ô∏è Effort Estimates

Estimate how long tasks will take.

** ‚ö†Ô∏è Setting Effort

Add an EFFORT property:

#+BEGIN_SRC org
,* TODO Review document
:PROPERTIES:
:EFFORT: 1:30
:END:
#+END_SRC

| Speed Key | Action                               |
|-----------+--------------------------------------|
| e        | Set effort estimate                  |

** ‚ö†Ô∏è Effort Format

Use H:MM format:
- 0:30 - 30 minutes
- 1:00 - 1 hour
- 2:30 - 2 hours 30 minutes

* ‚ö†Ô∏è Best Practices

** ‚ö†Ô∏è Task Granularity

Make tasks actionable:

#+BEGIN_SRC org
,* TODO "Write documentation"           ; Too vague
,* TODO Write introduction section      ; Better - specific and actionable
#+END_SRC

** ‚ö†Ô∏è Use NEXT for Clarity

Always have at least one NEXT task to know what to work on:

#+BEGIN_SRC org
,* Project: New Feature
,** DONE Design architecture
,** NEXT Implement core module           ; ‚Üê Clear next step
,** TODO Write tests
,** TODO Deploy to staging
#+END_SRC

** ‚ö†Ô∏è Regular Review

Periodically review tasks to:
- Move stale TODO items to HOLD or CANCELLED
- Update WAITING items
- Identify next actions

** ‚ö†Ô∏è Tags vs Hierarchy

Use hierarchy for project structure, tags for cross-cutting concerns:

#+BEGIN_SRC org
,* Work                                                       :work:
,** Project A
,*** TODO Task requiring phone                                :@phone:
,*** TODO Task requiring computer                             :@computer:
,** Project B
,*** TODO Another phone task                                  :@phone:
#+END_SRC

Now you can filter by @phone to see all phone tasks across projects.
