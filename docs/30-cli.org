* Introduction

Scimax VS Code includes a command-line interface (CLI) that provides access to many features without needing to open VS Code. This is useful for:

- Batch operations (exporting multiple files)
- Integration with other tools and scripts
- Quick lookups from the terminal
- CI/CD pipelines for documentation

The CLI reads from the same SQLite database that the VS Code extension builds, so you get consistent results between the editor and command line.

* Installation

** From the Extension

If you have the extension installed, the CLI is included. After building the extension:

#+begin_src bash
# From the extension directory
npm run compile

# Run directly
node out/cli/index.js --help

# Or use npx (if in the extension directory)
npx scimax --help
#+end_src

** Global Installation

To use ~scimax~ from anywhere:

#+begin_src bash
# From the extension directory
npm link

# Now available globally
scimax --help
#+end_src

To unlink later:
#+begin_src bash
npm unlink scimax-vscode
#+end_src

* ✅ Configuration
CLOSED: [2026-04-12 Sun 19:48]

The CLI automatically uses the same database as the VS Code extension. It looks in VS Code's global storage location:

| Platform | Database Path                                                                                   |
|----------+-------------------------------------------------------------------------------------------------|
| macOS    | =~/Library/Application Support/Code/User/globalStorage/jkitchin.scimax-vscode/scimax-db.sqlite= |
| Linux    | =~/.config/Code/User/globalStorage/jkitchin.scimax-vscode/scimax-db.sqlite=                     |
| Windows  | =%APPDATA%/Code/User/globalStorage/jkitchin.scimax-vscode/scimax-db.sqlite=                     |

The database is created and maintained by VS Code when you open a folder with org files. The CLI reads from and writes to this same database.

** Override Database Path

You can override the database path with the =--db= flag:

#+begin_src bash
scimax db stats --db /path/to/custom.db
scimax search "query" --db /path/to/custom.db
#+end_src

** Custom Configuration (Advanced)

For advanced setups, you can create =.scimax/config.json= in your project directory to specify a custom database path:

#+begin_src json
{
  "dbPath": "/custom/path/to/scimax.db"
}
#+end_src

The CLI checks for this file only if the VS Code global storage database is not found.

** Export

Export org files to HTML, LaTeX, or PDF.

#+begin_src bash
# Export to HTML
scimax export paper.org --format html

# Export to LaTeX
scimax export paper.org --format latex

# Export to PDF (requires pdflatex/latexmk)
scimax export paper.org --format pdf

# Export to Beamer slides (.tex)
scimax export slides.org --format beamer

# Compile Beamer slides to PDF
scimax export slides.org --format beamer-pdf

# Specify output file
scimax export paper.org --format html --output build/paper.html

# With bibliography
scimax export paper.org --format html --bib references.bib

# With specific CSL style
scimax export paper.org --format html --bib refs.bib --csl ieee
#+end_src

The export command automatically looks for bibliography files in the same directory (=refs.bib=, =references.bib=, or =<filename>.bib=).

** ✅ Tangle
CLOSED: [2026-07-11 Sat 09:08]

Extract source blocks to their =:tangle= target files (org-babel tangling). Each block with a =:tangle <filename>= header argument is written to that file, with multiple blocks targeting the same file concatenated in document order. Blocks with =:tangle no= (or no =:tangle= header) are skipped. Noweb references (=<<name>>=) are expanded by default for blocks marked =:noweb yes= or =:noweb tangle=.

#+begin_src bash
# Tangle all blocks in a file to their :tangle targets
scimax tangle setup.org

# Only tangle specific named blocks (matches #+NAME: of the block)
scimax tangle setup.org --only init,config

# Disable noweb reference expansion
scimax tangle setup.org --no-noweb

# Structured JSON output (files written, block/line counts, errors)
scimax tangle setup.org --json
#+end_src

Relative =:tangle= paths are resolved against the directory of the source file, and parent directories are created as needed. Note that =--only= filters *named* blocks; unnamed blocks with a =:tangle= target are always tangled.

Citation-related operations.

*** Extract Citations

List all citation keys used in a document:

#+begin_src bash
scimax cite extract paper.org

#+end_src

  doe-2020

  ...
#+end_example

*** Check Citations

Validate citations against a bibliography file:

#+begin_src bash
scimax cite check paper.org --bib references.bib
#+end_src

Output:
#+begin_example
Citation Check: paper.org

  Citations found: 12
  Bibliography entries: 15

MISSING (2 cited but not in .bib):
  - nonexistent-2020
  - typo-key

UNUSED (5 in .bib but not cited):
  - old-reference-2010
  - unused-entry
  ...
#+end_example

This is useful in CI pipelines to catch citation errors before publishing.

*** Convert Citation Syntax

Convert between org-ref v2, v3, and org-cite syntax:

#+begin_src bash
# Convert v2 to v3
scimax cite convert paper.org --from v2 --to v3

# Convert to org-cite
scimax cite convert paper.org --from v3 --to org-cite

# Modify file in place
scimax cite convert paper.org --from v2 --to v3 --inplace

# Output to new file
scimax cite convert paper.org --from v2 --to v3 --output paper-v3.org
#+end_src

*** List Bibliography

List entries in a BibTeX file:

#+begin_src bash
scimax cite list references.bib

# JSON output
scimax cite list references.bib --format json
#+end_src

** Journal

Open or create journal entries from the terminal.

#+begin_src bash
# Open today's journal (creates if needed)
scimax journal

# Open a specific date
scimax journal tomorrow
scimax journal yesterday
scimax journal friday
scimax journal --date "next monday"
scimax journal --date 2026-03-15
scimax journal --date +2d
scimax journal --date -1w

# JSON output (for scripting)
scimax journal --json
#+end_src

The journal directory is configured via =scimax.journal.directory= in VS Code settings. New entries use the default template (customizable via =scimax.journal.customTemplate= setting).

Supported date expressions: today, tomorrow, yesterday, day names (monday, fri), next/this day (next friday, this monday), relative (+2d, -1w, +3m), month+day (jan 15), ISO dates (2026-03-15).

** Project

Fuzzy-select and open a known project in VS Code.

#+begin_src bash
# Interactive fuzzy selection (uses fzf if available)
scimax project

# Pre-filter with a query
scimax project scimax

# List all known projects
scimax project --list

# Add a new project (creates directory if needed, opens in VS Code)
scimax project --add ~/projects/new-project

# JSON output
scimax project --json
#+end_src

Projects are pulled fresh from the scimax database each time. The extension registers projects automatically when you open folders in VS Code.

When =fzf= is installed, the interactive selection uses fuzzy matching. Otherwise, a numbered list is displayed.

*** ✅ Sync
CLOSED: [2026-07-11 Sat 09:08]

Bring the database into agreement with the configured scan roots and the filesystem in one pass: pick up new files, refresh changed files (mtime-based), refresh in-scope files that are stale, and prune entries for files that no longer exist.

Scan roots are unioned from:
1. =scimax.db.include= (VS Code settings)
2. =scimax.journal.directory=
3. =scimax.agenda.include=
4. NotebookManager projects (every project you have opened in VS Code, persisted in =notebooks.json=)

Files in the database that do not fall under any current root are still refreshed if they exist and their mtime has advanced; otherwise they are left alone.

#+begin_src bash
# Default daily driver
scimax db sync

# Preview what sync would do without changing the DB
scimax db sync --dry-run

# Show a line per file (in addition to the summary)
scimax db sync --verbose
#+end_src

Output:
#+begin_example
Database: ~/Library/Application Support/Code/User/globalStorage/jkitchin.scimax-vscode/scimax-db.sqlite

Scan roots (3):
  ~/Dropbox/org
  ~/Dropbox/journal
  ~/Dropbox/projects/scimax_vscode

sync complete in 1.4s
  +12 new
  ~5 updated
  -2 removed
  1834 unchanged
#+end_example

The summary uses these markers:

| Marker | Meaning                                                 |
|--------+---------------------------------------------------------|
| =+=    | New file added to the database                          |
| =~=    | Existing file re-indexed (mtime newer)                  |
| =*=    | Out-of-scope file refreshed because mtime advanced      |
| =-=    | File removed from the database (no longer on disk)      |
| =     | Unchanged                                                |

*** ✅ Clear
CLOSED: [2026-07-11 Sat 09:08]

Wipe the database. Use this before a full re-sync, after a schema change, or when troubleshooting suspected corruption. Requires confirmation.

#+begin_src bash
# Interactive confirmation
scimax db clear

# Skip the prompt
scimax db clear --yes

# Typical reset workflow
scimax db clear --yes && scimax db sync
#+end_src

Output:
#+begin_example
Database: ~/Library/Application Support/Code/User/globalStorage/jkitchin.scimax-vscode/scimax-db.sqlite
This will delete 1853 file(s), 12044 heading(s), 3201 source block(s).
Continue? [y/N] y
Database cleared.
Run "scimax db sync" to repopulate.
#+end_example

**** Cron Job Examples

Schedule automatic database updates:

#+begin_src bash
# Edit crontab
crontab -e

# Scan every hour
0 * * * * cd /path/to/project && scimax db scan ~/org-files

# Scan daily at 6am
0 6 * * * cd /path/to/project && scimax db scan ~/org-files >> /var/log/scimax-scan.log 2>&1

# Scan multiple directories every 30 minutes
*/30 * * * * cd /path/to/project && scimax db scan ~/work/notes && scimax db scan ~/personal/org
#+end_src

**** Command Comparison

| Command                | Use Case                                                  |
|------------------------+-----------------------------------------------------------|
| =scimax db sync=       | Daily driver: discover, refresh, and prune in one pass    |
| =scimax db clear=      | Wipe the database (followed by =sync= for a full rebuild) |
| =scimax db scan <dir>= | Ad-hoc scan of a single directory                         |
| =scimax db check=      | Read-only integrity report (no changes)                   |

- Use *sync* for routine maintenance, including the morning's first pass.
- Use *scan* for one-off directories outside the configured scan roots.
- Use *clear* + *sync* only when you genuinely need to start over.

The =rebuild= and =reindex= subcommands have been removed. Use =sync= (or =clear= + =sync=) instead.

* Limitations

The CLI has some limitations compared to the VS Code extension:

1. *Semantic Search*: Requires embeddings which need the embedding service configured in VS Code.

2. *Live Updates*: The CLI reads a snapshot of the database. Changes made to org files won't be reflected until you run =scimax db sync= or VS Code re-indexes them.

3. *Interactive Features*: No support for interactive features like completion or live preview.

4. *FTS Index*: =scimax db sync= populates the main tables but doesn't rebuild the FTS5 full-text index triggers. For full FTS support, use VS Code to create the initial database.
